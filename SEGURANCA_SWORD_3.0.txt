SWORD 3.0 - DOCUMENTO DE SEGURANCA
==================================

Objetivo
--------
Este documento descreve os avancos da versao 3.0 do Sword em relacao a
seguranca dos dados, controle de acesso, protecao contra invasoes e reducao de
risco contra injecoes externas.

Classificacao dos dados
-----------------------
Os dados do Sword devem ser tratados como informacao confidencial. Nomes de
dispositivos, IPs, hostnames, localizacoes, criticidade e historico de quedas
podem revelar a topologia da infraestrutura da empresa.

Avancos implementados
---------------------
1. Autenticacao obrigatoria
   - A API principal fica bloqueada sem login.
   - Na primeira execucao, o sistema exige criacao do administrador inicial.

2. Cargos e autorizacao
   - Administrador: controle total.
   - Operador: operacao de dispositivos, verificacoes e alertas.
   - Visualizador: somente leitura.
   - Rotas sensiveis retornam 403 quando o cargo nao permite a acao.

3. Senhas protegidas
   - Senhas nao sao salvas em texto puro.
   - O backend usa PBKDF2-SHA256 com salt individual.
   - Cada usuario tem hash proprio.

4. Sessoes protegidas
   - Sessao via cookie HttpOnly.
   - Cookie com SameSite=Strict.
   - Expiracao configuravel.
   - Logout revoga a sessao no servidor.

5. Protecao anti-CSRF
   - Acoes POST, PUT e DELETE exigem token X-CSRF-Token.
   - O token e vinculado a sessao.
   - Requisicoes de escrita sem token valido sao bloqueadas.

6. Rate limit de login
   - Tentativas falhas sao limitadas por email e origem.
   - Reduz risco de brute force.
   - Tentativas bloqueadas sao registradas na auditoria.

7. Headers defensivos
   - Content-Security-Policy para reduzir risco de XSS.
   - X-Frame-Options DENY contra clickjacking.
   - X-Content-Type-Options nosniff.
   - Referrer-Policy no-referrer.
   - Permissions-Policy desabilitando camera, microfone e geolocalizacao.

8. Validacao de entrada
   - IP/hostname com caracteres restritos.
   - Validacao de email, cargo, status e configuracoes numericas.
   - Limite de corpo HTTP em 1 MB.
   - O ping usa API nativa do .NET, nao comando de shell montado com entrada do usuario.

9. Auditoria
   - Login, falha de login, setup, criacao/edicao/exclusao de usuarios,
     operacoes em dispositivos, resolucao de alertas, alteracao de senha,
     exportacao e backup sao registrados.
   - Administradores possuem tela de auditoria.

10. Backups e exportacao
   - Administradores podem gerar backup local.
   - Exportacao e controlada por cargo.
   - Operacoes de exportacao sao auditadas.

11. Configuracoes de seguranca
   - Sessao, rate limit, CSRF, retencao de auditoria, retencao de eventos,
     timeout e tentativas de monitoramento sao configuraveis.

12. Identidade visual profissional
   - Nome Sword.
   - Icone de espada em SVG local.
   - Tipos ampliados de dispositivos com icones visuais.

Riscos que ainda exigem cuidado no ambiente
------------------------------------------
1. HTTPS
   - Em producao, nunca use login em HTTP puro.
   - Execute o Sword atras de um reverse proxy com HTTPS, como Nginx, Caddy,
     IIS ou Cloudflare Tunnel/Zero Trust.

2. Arquivo data/store.json
   - O banco local contem dados sensiveis.
   - Restrinja permissoes NTFS para o usuario que executa o Sword.
   - Nao compartilhe a pasta do projeto em rede aberta.
   - Nao envie este arquivo em mensagens, tickets ou repositorios publicos.

3. Backups
   - A pasta backups tambem contem dados sensiveis.
   - Proteja backups com permissao de arquivo e, idealmente, criptografia.
   - Armazene copia offline ou em cofre corporativo.

4. Acesso de rede
   - Por padrao, o servidor escuta em 127.0.0.1.
   - Para acesso por outros computadores, use VPN ou reverse proxy autenticado.
   - Evite expor a porta diretamente na internet.

5. Banco de dados
   - Para empresa real, planeje migrar de JSON para PostgreSQL ou SQLite
     protegido por backup e permissao adequada.

6. MFA
   - Para maior seguranca, adicione MFA para administradores.
   - Opcoes futuras: TOTP, Google Login, Microsoft Login ou SSO corporativo.

7. Segredos
   - Client secrets, tokens e senhas de integracao devem ficar fora do codigo.
   - Use arquivo de configuracao protegido ou variaveis de ambiente.

8. Hardening do servidor
   - Mantenha Windows atualizado.
   - Rode o Sword com usuario sem privilegio administrativo.
   - Use firewall local permitindo somente origens confiaveis.
   - Ative antivirus/EDR corporativo.

9. Logs e auditoria
   - Revise auditoria periodicamente.
   - Investigue falhas repetidas de login e bloqueios CSRF.
   - Exporte logs para SIEM futuramente.

Ambiente recomendado
--------------------
Uso local:
  http://localhost:8787
  Bom para teste, desenvolvimento e apresentacao controlada.

Uso interno seguro:
  VPN + HTTPS + firewall + usuario sem privilegio + backup protegido.

Uso corporativo:
  HTTPS obrigatorio, PostgreSQL/SQLite protegido, MFA para admins, backups
  criptografados, auditoria revisada e monitoramento do proprio servidor Sword.

Resumo de seguranca
-------------------
A versao 3.0 e muito mais segura do que as versoes anteriores e ja possui
controles reais de autenticacao, autorizacao, sessao, CSRF, rate limit,
auditoria e headers defensivos. Mesmo assim, nenhuma aplicacao e absolutamente
inviolavel. A seguranca final depende tambem do ambiente onde o Sword sera
executado: HTTPS, permissoes de arquivo, firewall, backup seguro e acesso
restrito sao obrigatorios para uso profissional.
